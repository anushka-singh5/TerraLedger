Process
How it works — end to end

Verification gates the mint. Credits are fungible. Retirement is permanent and tracked per project.

1
Submit + AI verify 5 modules · score ≥ 70
Developer submits GPS polygon, deed PDF and tonnage claim. Five AI modules run in sequence: GPS duplicate detector, deed forensics (OCR + ELA forgery check), anomaly model trained on Verra registry data, real NASA FIRMS satellite check and a Llama-3 plain-English audit. GPS overlap >20% or forged deed = instant hard-block, logged on-chain forever. Score below 70 = no mint.
2
Oracle mints TLCERT + TCC soulbound proof + fungible credits
On pass, the M-of-N oracle mints two things directly to the developer's wallet: a TLCERT soulbound NFT (non-transferable verification proof, bound to the project ID forever) and TCC ERC-20 tokens (1 TCC = 1 tonne CO₂, tradeable). TLCERT cannot be sold — it stays with the developer as permanent on-chain proof the project cleared all AI checks.
3
List & trade TCC QUSDC · per-tonne pricing · projectId stored on-chain
Developer lists TCC at a per-tonne QUSDC price on the marketplace. Every listing stores the source projectId on-chain — buyers always know which verified project they're buying credits from, even after credits change hands. No anonymous TCC floating around the market.
4
Retire TCC → TLRET certificate permanent offset + PDF download
Any TCC holder calls retire(). Tokens burn permanently against the correct project ID. A TLRET soulbound certificate auto-mints to the retiree's wallet (amount, projectId and block timestamp stored on-chain), and a PDF retirement certificate is instantly downloaded — with tx hash and full details. One retire call = one certificate. No double-counting possible.
5
TLCERT auto-offsets no button · no re-minting · tamper-proof
The contract tracks TCC issued vs. retired per project ID. When every TCC from a project is burned, the TLCERT automatically flips to Fully Offset — no manual action, no admin call. Since the project ID is locked at mint and GPS coordinates are registered on-chain, the same land parcel cannot generate new credits once fully offset.

Frequently Asked Questions
Everything about how verification works, what documents you need, how tokens are minted, and what lives on-chain.

Each module runs independently. Failing any hard-fail condition kills the submission immediately, regardless of other scores.

  • GPS Duplicate Detector · 30 pts — your polygon is checked against every approved project using on-chain bounding boxes as a pre-filter, then Shapely geometry for exact overlap. >20% overlap = hard-reject.
  • Ownership Document Parser · 25 pts — OCR extracts owner name and GPS from your deed, Error Level Analysis checks for digital tampering, and the document SHA-256 is compared against every previously-used document hash stored on-chain.
  • Carbon Anomaly AI · 25 pts — an Isolation Forest model trained on real Verra registry data flags statistically implausible tonnes-per-hectare claims vs. comparable verified projects.
  • NASA Satellite Check · 20 pts — live NASA FIRMS API query for fire or deforestation events over your polygon in the last 90 days. Recent burning = no credit.
  • Llama 3 Audit · no score — produces a plain-English audit report summarising all module results, pinned to IPFS and linked permanently to your TLCERT NFT.

Minimum 70 / 100 to mint. Scoring 69 or below means no NFT, no TCC — but you can fix the issue and resubmit. A low score is not a fraud flag.

Hard-fails are different: GPS overlap >20%, forged document, reused document, unreadable file, or wrong file type are instant rejections that do not depend on the total score. These are logged on-chain.

💡A failed submission does not block your wallet — only a confirmed fraud flag does. You can resubmit as many times as needed with corrected data.

The oracle is the on-chain bridge between the AI backend and the mint function. Instead of trusting a single backend result, the CarbonOracle contract requires a configurable number of independent validator signatures (M of N) before it will call mint. No single server can fake a credit.

On the current deployment the threshold is set by the contract owner. The M-of-N design means adding more validators later raises the bar without redeploying the token contract.

Yes — for score failures and most hard-fails. What you can fix and resubmit:

  • Score below 70 — fix the underlying issue (better deed scan, adjusted polygon, etc.) and resubmit
  • Unreadable document — re-scan at ≥300 DPI and resubmit
  • Project ID not in document — use a deed that references the correct parcel number
  • GPS mismatch — align your polygon with the coordinates in the deed

What you cannot resubmit: GPS overlap with an existing approved project (the land is already registered), or a wallet that has been permanently fraud-flagged on-chain.

Only: PDF, JPG, JPEG, PNG, TIFF, TIF. The type check happens before any AI analysis — everything else is rejected instantly with an INVALID_FILE_TYPE flag.

This means Word documents, spreadsheets, text files, screenshots saved as WebP or GIF, and ZIP archives are all blocked at the gate. The accepted types represent scanned physical deeds and title documents — the only sources that can meaningfully prove land ownership.

A generic ownership deed could legally belong to any land. By requiring the Project ID to appear in the deed text, we ensure the document was specifically prepared for this submission and hasn't been recycled from a different project.

The check is fuzzy — it handles spacing and separator variations (e.g., "Khasra No. 123/B" matches "Khasra 123/B"). But if the parcel number genuinely doesn't appear in the document, verification returns a PROJECT_ID_MISMATCH hard-fail.

Most common reasons — check these in order:

  • Scan quality — resolution below ~150 DPI makes OCR unreliable. Re-scan at 300 DPI or higher
  • Project ID missing — the parcel/khasra/survey number you entered must appear somewhere in the deed text
  • GPS mismatch — if GPS coordinates appear in the deed and are more than 50 km from your submitted polygon, it's a hard-fail
  • Authenticity below 30% — Error Level Analysis detected compression artefacts consistent with digital editing. Use the original scan, not a re-saved or screenshotted version
  • Not a deed — the document must contain land registry keywords or a registration number. A utility bill or ID card won't pass even if it has your name and GPS

Hard-fail, permanently logged on-chain. The SHA-256 hash of every accepted document is stored in the oracle contract at first submission. Any re-submission of the same file — even with metadata changed — is detected and flagged DOCUMENT_REUSE.

Forgery is detected via Error Level Analysis (ELA): digitally-edited images show JPEG re-compression artefacts in the altered regions. If ELA confidence exceeds the threshold, the document is flagged FORGED_DOCUMENT.

⛔Both flags result in a permanent on-chain fraud record against the submitting wallet. The wallet is blocked from future submissions. This cannot be undone — it is immutable on QIE Mainnet.

You don't — by design. A fraud flag written to the CarbonOracle contract is permanent and immutable. Blockchain state cannot be rolled back. The flagFraud() function is intentionally one-way with no corresponding un-flag function.

This is a core security guarantee: if fraud flags could be cleared by an admin, the system would have a single point of trust. An unforgeable on-chain record that permanently associates fraudulent behaviour with a wallet is the only design that can't be gamed.

🔍Every fraud flag is publicly readable — wallet address, project ID, reason, block timestamp. Anyone can query the contract and see the full history.

The minting flow is fully gated — no human can manually trigger a mint:

  • You submit a project; the AI backend runs all five modules
  • If score ≥ 70 and no hard-fails, the backend calls submitVerification() on the CarbonOracle contract with the per-module scores and document hash
  • Once the required oracle validators attest, the oracle calls mint() on CarbonCreditToken — exactly the claimed number of TCC go directly to your wallet
  • Simultaneously, a TLCERT soulbound NFT is minted as your permanent verification certificate

1 TCC = 1 tonne CO₂ sequestered. TCC has zero decimals — you can't have half a tonne.

  • TLCERT (ERC-721, soulbound) — the verification certificate. Non-transferable, tied to your wallet and project ID. Stores AI score, per-module breakdown, document hash, IPFS audit link, and current project status. You get exactly one per verified project.
  • TCC (ERC-20, transferable) — the actual carbon credit. 1 TCC = 1 tonne CO₂. Fully fungible. Can be held, listed on the marketplace, sold, and eventually burned (retired). This is the token that represents the carbon offset.
  • TLRET (ERC-721, soulbound) — the retirement certificate. Minted to the wallet that calls retire(). Non-transferable proof that specific tonnes of CO₂ were permanently burned. Each retirement produces one TLRET.

When you call retire(), two things happen simultaneously on-chain:

  • The specified amount of TCC is permanently burned — destroyed with no possibility of recovery
  • A TLRET soulbound NFT is minted to your wallet, recording: project ID, tonnes burned, your wallet address, and the block timestamp

From the UI you can also download a PDF retirement certificate that includes the same details plus the transaction hash — a human-readable document backed by an immutable on-chain record anyone can verify on QIE Explorer.

📄The retirement is on QIE Mainnet (chain 1990). The transaction hash in your certificate can be pasted into the QIE Explorer to independently confirm the burn.

The TLCERT for that project automatically updates its status to Fully Offset on-chain. No admin action, no manual update — CarbonCreditToken tracks total minted vs. total retired per project, and flips the certificate when they match.

This means a corporate buyer who retires the entire supply of a project's TCC gets a cryptographic proof that 100% of the verified credits from that land area have been permanently removed from circulation.

The marketplace settles in QUSDC — QIE's stablecoin. Sellers set a price per TCC in QUSDC. Buyers pay QUSDC; TCC goes to the buyer; QUSDC goes to the seller minus a protocol fee.

The fee is configurable by the contract owner and is capped at 10% (1000 bps) by the MAX_FEE_BPS constant in the TCCMarketplace contract. The fee goes to a treasury address, also set by the owner. Both are readable on-chain — there are no hidden deductions.

  • ProjectRegistry — GPS polygon GeoJSON, area, claimed tonnes, bounding box, status. Fully on-chain.
  • CarbonOracle — total score, per-module scores, document SHA-256 hash, fraud log. All on-chain.
  • CarbonCreditToken / TLCERT — total minted, total retired per project, certificate status. On-chain.
  • IPFS — the Llama 3 audit report. The CID is linked to the NFT's tokenURI so it's permanently pinned via Pinata.

Nothing verification-critical is off-chain-only. The backend is the gate — the chain is the record.

Two-layer architecture — cheap on-chain pre-filter + precise off-chain geometry:

  • Every approved project stores a bounding box (min/max lat/lng) in ProjectRegistry as four int64 values (coordinates × 1e6 to avoid floats)
  • On submission, the backend calls findOverlaps() on-chain — this returns all approved projects whose bounding boxes intersect yours
  • For each candidate, Shapely runs an exact polygon intersection off-chain to compute the real overlap percentage
  • >20% overlap = hard-reject (OVERLAP_HARD_FAIL). 5–20% = warning flag added, but you can still pass if scores are high enough

The bounding-box guard is publicly queryable — anyone can call findOverlaps() directly to see if their land would conflict before submitting.

The CarbonOracle contract emits a FraudFlagged event and writes the record to chain:

  • Project ID that was submitted
  • Wallet address of the submitter
  • Failure reason string (e.g., "Forged document", "GPS overlap with project 0x…")
  • Block timestamp — permanent, immutable, publicly readable forever

Additionally, the oracle marks that project ID as FraudFlagged in ProjectRegistry — status that blocks any future action on that project. You can see all fraud events in the Fraud Activity Log on the home page, read live from the contract.
